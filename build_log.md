Hints are sufficiently convincing. However, Socratic behaviour is lost at some point, and it starts behaving like traditional ChatGPT. It will go back to Socratic for the next prompt if asked (but will only last 1 prompt). It also randomly changes the answer structure. (Note: this feedback was given when project used gpt-4o for tutor and gpt-40-mini for assessor. It is unknown if this behaviour is model dependent or not.)
  - The original "Full Detailed Answer Mode" section let the model switch out of Socratic mode on its own judgement ("repeated guided attempts have failed and a complete worked example is educationally justified"), and the Default Response Structure had an escape hatch ("...unless another format is clearly more suitable"). Neither section told the model to snap back afterwards — so once it free-formed one answer, its own prior turn became the exemplar it continued from, and only an explicit ask would (briefly) restore Socratic mode. I rewrote both sections in prompts/tutor.md: the full-answer trigger now requires an explicit request only (stalled attempts get an offer, not an automatic switch), the format escape hatch is gone, and there's now a bolded reversion rule: return to the Default Response Structure immediately after a full answer, even mid-problem.

- Crash on Windows console encoding (utils/tutor/interface.py:1-16) — the real reason "the bot output wasn't printed" persisted even after the vector store fix. 
  - Several print() debug statements echo raw model output (including the Unicode minus sign −, common in generated equations) straight to the console. Windows defaults stdout to the cp1252 codec, which can't encode that character, so print() raised UnicodeEncodeError mid-response — before the answer was ever added to chat history or rendered. I reconfigured stdout/stderr to UTF-8 with errors="replace" at module import time, so a debug print can never take down a real answer again.

- Does the tutor reload chat history per competency/topic?
  - All messages for a module (across every topic discussed that session) live in one list, st.session_state["messages_module_{id}"]. The UI shows this whole list — that's why you see earlier topics' Q&A still sitting above the new one after a transition. But what the model actually receives is different: on every topic transition, TutorState.set_topic_cutoff_index() (utils/tutor/state.py:76) records the current message count as a cutoff. get_conversation_context() only sends messages after that cutoff, capped at the last 5 — so the model only ever sees the current topic's exchanges, never prior topics'. The opening question for each new topic is generated via a standalone API call (build_initial_topic_prompt, no previous_response_id) — so each topic starts the model with zero prior turns, not chained to the previous topic's conversation. Within a topic, turns are chained via previous_response_id for continuity. None of this persists across a logout/refresh except via mongodb/logger.py's conversation log, which is replayed (load_logged_conversation_messages) only for a topic that was already fully logged before.
- Can we prompt cache the tutor competency prompt so that it saves token usage and just append the relevant course index stuff.
  - It's already possible and free — OpenAI's Responses API caches automatically based on matching the identical leading prefix of instructions+input across calls, no manual index or cache key needed. But utils/tutor/interface.py:492-500 currently defeats it: it builds system_prompt by prepending the per-topic bits (Current Topic: ..., description, learning outcomes) before the static tutor.md content. Since prefix-caching only matches from byte 0, and the topic name changes every call, the cache never hits even though most of the ~1-2k token block is identical every time. Swapping the order — static tutor.md first, then a ## Current Topic section appended after lets the bulk of it get cached automatically and cut cost/latency on every turn.
- Tutor bot returns blank response. There are multiple issues that have already been fixed regarding this.
  - GPT-5-mini is a reasoning model. Before it writes the visible reply, it generates internal "reasoning tokens" that are billed as output tokens even though the student never sees them. Neither API call in interface.py sets a reasoning effort level or max_output_tokens, so the model is left to decide on its own how much to think. For this system prompt specifically (Socratic pedagogy rules + strict Markdown/LaTeX formatting + function-calling + Australian spelling, all at once), that's a lot to juggle, so it can burn a large chunk of its token budget reasoning about the reply before writing any of it. If it runs out mid-reasoning, you get: zero visible text, no function call, nothing. Not because the request was cheap, but because the "thinking" phase ate the whole allowance. To fix the source of the issue, pass reasoning={"effort": "low"} (or "minimal") and an explicit max_output_tokens on these calls, guaranteeing room is reserved for the actual reply.
